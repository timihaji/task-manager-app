What this is — and isn't

Goal. Make a project feel like a place you go, not a row that expands. Concentrate everything you need to think about a project — the brief, the active work, the metrics, the decisions, the history — in one workspace. Everything else (Inbox, Stack, Week) keeps working unchanged.

Non-goals. Custom fields per project, an automation/rules engine, sub-projects (project nested in project), and per-project custom statuses. The board uses a fixed three-status model (To do / Doing / Done) for v1; per-project columns are a v2 conversation.

The single load-bearing decision: a dedicated full-page view (view = {type:'projectDetail', id}), not a drawer or modal. Already aligned with how the existing view-routing works in App.jsx.

Data model

Seven new project-level fields plus one subtask-level field. Migrate via a single Supabase migration; map snake_case ↔ camelCase in db.js.

statustext'active'Project lifecycle: planned · active · on_hold · done · abandoned. Project-level only (NULL on regular tasks).

Decisions and links — JSONB on the project

Decisions and links don't justify their own tables for v1. Store as JSONB arrays on the project row:

-- on the project task row
decisions jsonb DEFAULT '[]'::jsonb
-- shape: [{ id, at: timestamptz, body: text, by?: text }]

links jsonb DEFAULT '[]'::jsonb
-- shape: [{ id, url: text, label?: text }]

Derived (no schema change)

  • Health — at_risk if any subtask overdue or blocked or stale > 7 days; off_track if any subtask overdue > 3 days; else on_track. Computed in a useProjectStats hook, not stored.
  • % done — count of subtasks with subtask_status='done' ÷ total. Computed each render.
  • Velocity — completions over the last N days vs. the prior N. Computed from the existing completedAt timestamps.
  • Time spent — sum of timeEstimate on done subtasks; "estimate" sums all subtasks.
Realtime: the existing Supabase subscribeTasks stub in db.js is unwired (per HANDOFF.md). This plan does not require it — every metric is derived client-side from already-loaded tasks. If realtime ships separately, the metrics auto-update.

Phased rollout

Each phase is independently shippable. Stop after any phase and the app still works — earlier phases just have fewer tabs.

1

Skeleton + routing~2 days

Get to the destination first. No interactivity inside.

  • Run the schema migration; extend makeTask + db.js mappings.
  • Build ProjectDetailView.jsx shell + ProjectToolbarStack.jsx.
  • Wire {type:'projectDetail', id} in App.jsx; click handler in TaskCard; Esc → lastView.
  • All tabs present in the strip; only Board renders content (read-only kanban via existing data).
  • useProjectStats implemented + driving the metric bar.
2

Board interactions~3 days

The kanban metaphor only works if you can move cards.

  • Extract useDragReorder from StackView; preserve the dragover-on-source guard and ≥80ms defer rule (per CLAUDE.md).
  • Drag-between-columns updates subtaskStatus; if 'done', set done=true + completedAt.
  • Quick-add bar at the bottom of each column; n focuses current column.
  • Click card → existing drawer.
3

Brief + Decisions~2 days

Make the project feel like more than a folder.

  • ProjectBriefTab: outcome edit-on-click, body textarea.
  • ProjectDecisionsTab: append-only, JSONB on project.
  • Header chrome shows outcome as a pinned strip (above board) when set.
4

List + Activity~2 days

Sort/scan power-tools and the receipts.

  • ProjectListTab: sortable subtask table, j/k/x keys.
  • ProjectActivityTab: aggregate activity jsonb across project + subtasks.
  • Sidebar: Pinned projects section (uses pinned_at).
5

Polish + Files~3-4 days

Sand off the rough edges; ship what's left.

  • Markdown rendering for Brief (decide library; marked + DOMPurify is the obvious cheap pick).
  • ProjectFilesTab: links MVP (paste URL → favicon row).
  • Mobile-friendly: stack toolbars vertically, scroll-snap kanban columns, touch-drag tested.
  • Keyboard: g b/l/r/d/a/f to switch tabs; esc to back.
  • Cover/icon/accent picker in toolbar's gear menu.

Critical implementation notes

Things that have bitten this codebase before, and the schema move that needs care.

Drag bug #1 — source mutation aborts drag

Setting display:none / removing the source synchronously after dragstart cancels the drag in Chromium. When extracting useDragReorder, defer any visibility change ≥80ms. CLAUDE.md covers this in detail.

Drag bug #2 — element under cursor flickers

If the source slides under the cursor on hover, dragover-on-source triggers a state reset → flicker loop at 60Hz. Preserve the early-return on dragover-on-source from StackView when extracting.

Schema migration order

Apply 0002_project_fields.sql in dev Supabase first. Verify MigrateFromLocal.jsx still works (or extend it to carry the new fields). RLS policies remain unchanged — new columns inherit auth.uid() = user_id.

Naming clash — "project"

task.project is a context (LIFE/WORK/HOME), not a project container. UI already says "Location" in some places — finish that rename pass as part of phase 4 to prevent every new project feature from inheriting the ambiguity. Data-model rename is deferred.

Open questions

Decisions to make during implementation, not before. Each is small enough to flip without re-architecting.

  • When a project flips to Done, do we auto-complete remaining subtasks, or surface a "mark all done?" prompt? Lean: prompt, never auto.
  • Pin order in sidebar — manual reorder, or last-opened-first? Lean: manual; explicit beats clever.
  • Does the quick-add bar accept /commands (e.g. /today, /p1) inline in v2? Defer; ship plain text first.
  • If a card is dragged from Doing to Done while it has open subtasks-of-subtasks (we don't currently nest, but might), what happens? Out of scope — sub-projects are explicitly rejected.
  • When does Velocity (+8%/wk) show vs. hide? Need a baseline of ≥ 7 days of activity to mean anything. Lean: hide chip if < 7d of data.
